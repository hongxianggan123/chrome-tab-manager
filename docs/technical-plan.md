# 技术方案

## 技术选型

当前实现采用：

- Chrome Extension Manifest V3
- TypeScript
- React 作为侧边栏 UI
- Vite 作为前端构建工具
- shadcn/ui 作为基础组件系统
- Chrome Side Panel API 作为主要入口
- Chrome Tabs / Windows / Tab Groups API 作为标签页数据来源
- Chrome Storage API 保存用户配置和 URL 级状态

## 扩展结构

当前代码结构分为四层：

- 后台服务层：运行在 MV3 service worker 中，监听标签页、窗口、标签组、存储变化。
- 领域状态层：维护标签实例、标签记录、重复组、归档状态和分组状态。
- UI 层：侧边栏展示分组列表、搜索、过滤、批量操作栏和单项操作。完整 URL 查看入口暂不在当前 UI 中展示，后续单独设计。
- 持久化层：封装 Chrome Storage，负责用户配置和 URL 级状态读写。

## 关键 Chrome 能力

MVP 已使用的 Chrome API：

- `chrome.tabs`：查询、激活、关闭、监听创建/更新/激活/关闭。
- `chrome.windows`：查询窗口、聚焦窗口、创建窗口。
- `chrome.sidePanel`：提供侧边栏入口。
- `chrome.storage`：保存归档记录和分组折叠状态。

后续阶段需要评估的 Chrome API：

- `chrome.tabGroups`：读取 Chrome 原生标签组元信息。
- `chrome.alarms`：周期性检查休眠自动归档。
- `chrome.processes`：按需读取标签页对应浏览器进程及进程级内存信息。

后续实现时需要以实际 Manifest V3 权限为准，避免过度申请权限。

内存指标不应默认持续轮询。需要通过按需刷新、可选列或详情面板读取。`chrome.processes` 提供的是进程级信息，多个标签页可能共享进程，因此 UI 文案不能把它表达为单个标签页的精确独占内存。

## 状态模型

运行时状态分为三类：

- 浏览器实时状态：当前普通窗口中的标签实例、窗口信息、Chrome 原生标签组信息、音频播放状态、原生置顶状态。
- 用户持久状态：归档标签记录、分组折叠状态；后续扩展为侧边栏置顶、置顶顺序、自定义分组规则、休眠自动归档设置。
- 派生状态：规范化 URL、重复组、标签清单分组、搜索和过滤结果；后续扩展为休眠判断。
- 按需诊断状态：内存指标等短期采样结果。

标签实例以 Chrome tab id 为运行时身份，但 tab id 不应作为长期持久身份。长期身份使用规范化 URL。

## URL 规范化

第一版规范化规则：

1. 解析原始 URL。
2. 保留 origin。
3. 保留 pathname。
4. 保留 search/query。
5. 丢弃 hash/fragment。

该规则已用于重复检测和归档；后续侧边栏置顶、自定义分组 URL 模式也必须复用同一规则，避免不同功能对同一 URL 得出不同身份。

需要识别特殊 URL，例如浏览器内部页面、扩展页面、本地文件 URL。特殊 URL 可展示和跳转，但归档、休眠自动归档和自动重复处理需要保守处理：不归档、不休眠自动归档，自动重复处理降级为提示。

## 分组计算

MVP 已实现完整 host 分组。后续自定义分组阶段使用以下优先级：

1. 如果命中自定义分组，使用第一个命中的自定义分组。
2. 如果未命中，使用完整 host 分组。

自定义分组由有序分组列表构成。每个分组可以包含多条匹配规则，组内任意规则命中即视为该分组命中。自定义分组和组内规则都需要支持启用/禁用。

自定义分组名称在当前 Chrome profile 内唯一。保存或重命名分组时需要校验名称冲突。

自定义分组优先级由用户手动排序决定。新建分组默认插入优先级列表顶部。

组内规则没有优先级，任意启用规则命中即可。UI 可以按创建顺序展示组内规则，但第一版不提供组内规则排序。

自定义分组规则支持：

- 完整 host 匹配
- URL 模式，即对规范化 URL 执行正则匹配

完整 host 匹配只匹配 URL 的 hostname，不包含端口，且大小写不敏感。需要区分端口的场景使用 URL 模式。

URL 模式第一版默认大小写不敏感，不提供正则 flags 配置。

第一版不支持页面标题匹配或标题正则。自定义分组规则只依赖 URL 相关信息。

需要在配置 UI 中处理无效正则。无效规则不应破坏整个标签清单。

## 重复处理

重复检测基于当前标签集合中的规范化 URL。重复处理模式有提示和自动两种。

第一版重复处理只响应用户打开或导航到新的普通网页标签实例，并在 URL 稳定后发现该规范化 URL 已有既有匹配实例。浏览器启动恢复、hash-only 变化、reload，以及同一标签页内导航到已有 URL 的复杂场景只触发 snapshot refresh 和重复计数更新，不创建待处理重复提示，也不执行自动移除。

默认目标选择规则：

- 排除新打开的重复标签实例，只在既有匹配实例中选择。
- 优先选择 `lastAccessed` 最大的既有实例。
- 如果访问时间不可用，使用当前稳定排序中的第一个既有实例。
- 提示模式和自动模式使用同一套默认目标选择规则。

自动模式的技术边界：

- 命中新打开标签页的规范化 URL 后，找到最近激活的既有匹配实例。
- 聚焦既有实例所在窗口并激活该标签页。
- 移除新打开的重复标签页。
- 正在播放音频或 Chrome 原生 pinned 的新重复标签页与普通标签页使用相同规则，不额外跳过关闭。
- 如果无法安全执行，应降级为提示。

提示模式默认通过侧边栏呈现，不使用系统通知。侧边栏提示展示在顶部控制区下方、标签清单上方，需要能展示默认跳转目标，并提供保留当前标签页和查看所有重复实例的动作。用户开启页面浮层并授予页面注入权限后，service worker 可在触发提示的新重复标签实例所在普通网页内注入页面浮层。页面浮层和侧边栏提示使用同一组动作语义。三个提示动作执行后都需要清理当前提示。用户选择跳转到已有标签页时，service worker 应先聚焦并激活目标既有实例，再移除触发提示的新重复标签实例；如果新重复标签实例已不存在或已不再匹配该提示，只执行可安全完成的部分并清理提示。

side panel 或页面浮层展示重复提示后应启动 30 秒 UI 生命周期；最后 5 秒在保留当前标签页按钮上展示倒计时。倒计时结束时，展示面向 service worker 发送提示关闭动作，service worker 清理对应待处理提示；该关闭动作不关闭标签页，但需要记录该 tabId 已在当前会话内处理，避免后续 snapshot refresh 为同一标签实例重新创建提示。

如果侧边栏未打开，且页面浮层未开启、未授权或当前页面不可注入，service worker 最多在 `chrome.storage.session` 保存最近一条待处理重复提示，并设置扩展 action badge/title 作为轻提醒；新的重复提示覆盖旧的待处理提示。侧边栏打开后读取并展示这条待处理提示。提示模式不应在没有用户手势时主动打开侧边栏；自动模式可以在后台直接执行跳转和移除新重复标签页。

页面浮层需要可选页面注入能力。实现时在用户开启设置后请求 `scripting` 与 `<all_urls>` optional host permission，并只在用户授权后的普通网页中按需动态注入；`<all_urls>` 不作为默认安装权限。页面浮层不通过 manifest 静态注册到所有页面，只在确实产生重复提示的新重复标签实例中注入。若授权被拒绝、注入失败、目标是特殊 URL 或浏览器限制页面，则降级为侧边栏待处理提示。若用户之后关闭或撤销页面浮层授权，service worker 应把 `duplicatePromptSettings.displayMode` 自动切回 `sidePanel`，并向侧边栏返回一次非阻塞提示。页面浮层代码不读取网页正文，不扫描 DOM，不保存页面内容，只挂载独立提示 UI 并通过 `chrome.runtime.sendMessage` 发送用户动作。

待处理重复提示属于运行时会话状态，不需要跨浏览器重启持久化。service worker 使用 `chrome.storage.session` 保存最近一条待处理提示和当前会话内已处理的 tabId 集合，避免 MV3 service worker 挂起后重复弹出。相关新重复标签页关闭、被自动移除、被用户选择保留或提示自动关闭后，对应提示应立即清理。用户选择保留当前重复标签页或提示自动关闭时，service worker 需要记录该 tabId 已在当前会话内处理；后续 snapshot refresh 不再为同一 tabId 重新创建提示，但不创建 URL 级豁免。浏览器重启后这些会话状态可以丢失。

## 休眠自动归档

休眠判断依赖扩展维护的最后激活时间。每次标签激活时更新该标签实例的最后激活时间。第一版只支持全局休眠阈值，不支持按自定义分组或完整域名单独配置阈值。

扩展首次观察到已有标签实例时，将首次观察时间作为该实例的最后激活时间初始化值。不要尝试推断安装前的真实激活时间，也不要在安装后立即把已有标签判定为休眠。

自动归档通过周期任务执行，默认关闭。执行时跳过：

- 保活实例
- 正在播放音频的实例
- Chrome 原生置顶实例

自动归档一个实例后，如果同一规范化 URL 没有其他打开实例，则创建归档状态；如果仍有其他打开实例，则只移除当前实例，不创建归档状态。

自动归档执行前不逐项提示用户。执行后需要写入近期自动归档运行时记录，并在侧边栏中展示。该记录用于解释自动行为，不作为完整历史或审计日志长期保存。

最近自动归档记录属于运行时会话状态，只保留最近 20 条，不跨浏览器重启持久化。

## 数据存储

MVP 使用 `chrome.storage.local`。由于产品范围限定在单个 Chrome profile 内，不使用 `chrome.storage.sync`，避免跨设备同步带来的容量限制、延迟和冲突。

MVP storage schema 见 [Storage Schema](./storage-schema.md)。运行时状态模型见 [Runtime State Model](./runtime-state-model.md)。service worker 事件流见 [Service Worker Flow](./service-worker-flow.md)。Chrome 权限策略见 [Chrome 权限最小集](./permissions-plan.md)。项目结构见 [项目结构方案](./project-structure.md)。构建与开发工具见 [构建与开发工具方案](./build-tooling-plan.md)。

当前 MVP 已持久化：

- 归档记录：按规范化 URL 保存标题、原始 URL、host、最近归档时间、归档次数、最后已知来源窗口。
- 视图偏好：普通分组折叠状态。搜索词不持久化，状态过滤只作为当前侧边栏会话状态。

后续持久化数据按逻辑拆分：

- 设置：重复处理模式、休眠阈值、休眠自动归档开关。
- 侧边栏置顶：按规范化 URL 保存是否置顶和手动排序。
- 自定义分组：分组列表、分组优先级、分组名称、分组启用状态、组内规则列表、规则启用状态。

保活是标签实例级状态，不跨实例生命周期。它可以保存在运行时状态中；如需应对 service worker 休眠，可短期持久化 tab id 映射，但标签关闭后必须清理。

第一版不做归档记录自动过期清理。归档记录只因用户打开归档项或显式删除归档记录而移除。

如果同一规范化 URL 的多个实例原始 URL 不同，归档记录使用最后被归档实例的原始 URL 作为恢复 URL。

归档记录的标题和 favicon URL 使用最后被归档实例的显示信息。打开中的标签实例展示实时标题和 favicon。标题和 favicon 只作为显示字段，不作为身份键。

## 权限和隐私边界

第一版不记录完整浏览历史，只保存产品必要状态。

隐身窗口不纳入管理范围。即使用户未来允许扩展在隐身模式运行，也不应默认混入主标签清单。

技术实现应避免保存页面内容、表单内容、滚动位置、登录态或网页内部状态。
